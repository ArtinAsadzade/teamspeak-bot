# نقشه راه کامل قابلیت‌های بات مدیریت TeamSpeak

این سند نقشه راه محصولی و فنی بات را به ماژول‌های مستقل تقسیم می‌کند تا قابلیت‌ها در یک فایل یا سرویس واحد مخلوط نشوند. همه پیام‌های قابل مشاهده برای کاربر باید فارسی، محترمانه، کوتاه و قابل فهم باشند.

## اصول معماری

- هر دامنه در مسیر مستقل زیر `src/features/<module>` پیاده‌سازی شود و فقط از طریق service، repository و event handler همان ماژول توسعه پیدا کند.
- وضعیت‌های ماندگار در Redis یا دیتابیس ذخیره شوند و کلیدها با prefix ماژول نام‌گذاری شوند.
- هر قابلیت با flag مستقل در `.env` کنترل شود و مقدار نهایی از `src/config/featureFlags.ts` خوانده شود.
- APIهای مدیریتی فقط پشت `x-api-secret` فعال باشند و اطلاعات حساس مثل رمز کانال، IP و uniqueId در لاگ‌ها redact شوند.
- تمام پیام‌های خصوصی، اعلان‌ها، خطاها و پیام‌های staff از catalog فارسی متمرکز در `src/features/messages/fa.ts` خوانده شوند.

## Flagهای پیشنهادی

| Flag | پیش‌فرض | ماژول | توضیح |
| --- | --- | --- | --- |
| `FEATURE_TEMP_CHANNEL_LIFECYCLE` | `true` | `tempChannels` | ساخت، تمدید، تغییر مشخصات و حذف خودکار چنل موقت |
| `FEATURE_TEMP_CHANNEL_OWNER_CONTROLS` | `false` | `ownerControls` | کنترل‌های مالک مثل kick، mute، ظرفیت و عمومی/رمزدار کردن |
| `FEATURE_SUPPORT_TICKETS` | `true` | `tickets` | تیکت چند لابی، انتقال خودکار، بستن و امتیازدهی |
| `FEATURE_ADMIN_API` | `true` | `admin` | API وضعیت، لیست‌ها، حذف دستی و پیام همگانی |
| `FEATURE_ANTI_SPAM_SECURITY` | `true` | `security` | rate limit، تشخیص جابه‌جایی سریع، blacklist و محافظت دستی |
| `FEATURE_AUTOMATION_RECOVERY` | `true` | `automation` | cleanup، بازیابی بعد restart، leader scan و health monitor |
| `FEATURE_WELCOME_AND_RULES` | `false` | `welcome` | خوشامد و قوانین سرور با پیام خصوصی |
| `FEATURE_AFK_SYSTEM` | `false` | `afk` | انتقال خودکار کاربران بی‌حرکت |
| `FEATURE_VIP_SYSTEM` | `false` | `vip` | زمان و ظرفیت بیشتر برای کاربران VIP |
| `FEATURE_DAILY_ADMIN_STATS` | `false` | `stats` | گزارش روزانه برای ادمین |
| `FEATURE_STAFF_AUDIT_LOG` | `false` | `auditLog` | لاگ رویدادهای مهم در چنل استف |

## فاز 1: User comfort

ماژول هدف: `src/features/tempChannels`

- ساخت چنل موقت با رمز تصادفی دقیقاً ۴ رقمی و ارسال رمز فقط با پیام خصوصی فارسی.
- تمدید زمان چنل با پیام خصوصی مالک و ثبت زمان جدید در metadata.
- تغییر اسم چنل توسط مالک با sanitize نام، محدودیت طول و جلوگیری از کلمات ممنوع.
- تغییر رمز چنل توسط مالک با رمز ۴ رقمی جدید یا انتخاب رمز توسط مالک در صورت معتبر بودن.
- قفل/باز کردن چنل توسط مالک از طریق password flag یا محدودیت join.
- دعوت کاربر به چنل با دستور پیام خصوصی یا API امن برای staff/admin.
- نمایش مشخصات چنل به مالک شامل نام، ظرفیت، وضعیت رمز، زمان تمدید، زمان حذف و کاربران حاضر.
- تکمیل catalog پیام فارسی برای موفقیت، خطا، عدم دسترسی، ورودی نامعتبر و محدودیت زمانی.

## فاز 2: Owner controls

ماژول هدف: `src/features/ownerControls`

- مالک چنل بتواند کاربر حاضر در چنل خود را kick کند، نه کاربران خارج از چنل.
- مالک بتواند کاربر را mute کند و مدت mute در Redis ذخیره شود.
- مالک بتواند ظرفیت چنل را در بازه مجاز config تغییر دهد.
- مالک بتواند چنل را عمومی یا رمزدار کند و نتیجه با پیام فارسی اعلام شود.
- جلوگیری از ساخت چند چنل همزمان برای یک نفر با mapping پایدار `ownerKey -> channelId`.
- audit داخلی برای همه عملیات مالک با actor، target، channelId و نتیجه عملیات.

## فاز 3: Support system

ماژول هدف: `src/features/tickets`

- پشتیبانی از چند lobby با `SUPPORT_LOBBY_CHANNEL_IDS` و routing بر اساس lobby ورودی.
- انتقال خودکار کاربر به چنل تیکت ساخته‌شده یا تیکت باز قبلی.
- نوتیف فارسی برای staff با نام کاربر، lobby، زمان شروع و لینک/شناسه چنل.
- بستن تیکت توسط staff مجاز با پیام فارسی برای کاربر و staff.
- لاگ مدت زمان پاسخ‌گویی از لحظه ساخت تا اولین ورود staff.
- امتیازدهی کاربر بعد از پایان تیکت با پیام خصوصی و ذخیره امتیاز.
- ذخیره تاریخچه تیکت در Redis/DB شامل وضعیت، زمان‌ها، staffها، امتیاز و خلاصه رویدادها.

## فاز 4: Admin management

ماژول هدف: `src/features/admin`

- پنل API برای دیدن وضعیت TeamSpeak، Redis، leader election و queueهای cleanup.
- endpoint لیست چنل‌های فعال ساخته‌شده توسط بات با مالک و metadata.
- endpoint لیست چنل‌های خالی در صف حذف با `emptySince` و زمان باقی‌مانده.
- حذف دستی چنل‌های ساخته‌شده توسط بات همراه با پاکسازی metadata.
- ارسال پیام همگانی به کل سرور، یک چنل یا یک کاربر با متن فارسی کنترل‌شده.
- لیست کاربران آنلاین با nickname، clientId، channelId و وضعیت‌های قابل انتشار.
- آمار تعداد کاربران آنلاین، چنل‌ها، چنل‌های موقت، تیکت‌های باز/بسته و خطاهای اخیر.

## فاز 5: Anti-spam / security

ماژول هدف: `src/features/security`

- محدودیت ساخت چنل بر اساس ownerKey، clientDbId، uniqueId و IP در پنجره زمانی قابل تنظیم.
- تشخیص جابه‌جایی سریع بین چنل‌ها و ثبت strike برای رفتار مزاحم.
- بلاک موقت کاربر مزاحم با TTL و پیام خصوصی فارسی در صورت امکان.
- blacklist برای `clientDbId`، `uniqueId` و IP با reload امن از API.
- لاگ کامل تخلفات شامل نوع تخلف، شناسه‌ها، زمان و action اعمال‌شده.
- محافظت از چنل‌های مدیریت‌شده در برابر تغییر دستی با reconcile دوره‌ای و بازگردانی تنظیمات مجاز.

## فاز 6: Automation

ماژول هدف: `src/features/automation`

- حذف خودکار چنل خالی بعد از `EMPTY_DELETE_DELAY_SEC` و لغو حذف در صورت بازگشت کاربر.
- پاکسازی metadata خراب Redis وقتی چنل در TeamSpeak وجود ندارد یا owner mapping ناسازگار است.
- بازیابی کامل بعد از restart با اسکن Redis و TeamSpeak و reconcile وضعیت‌ها.
- اسکن لابی‌ها بعد از Leader شدن تا کاربرانی که قبل از leader شدن وارد شده‌اند پردازش شوند.
- مانیتور سلامت اتصال TeamSpeak و Redis با metric، log و readiness دقیق.

## فاز 7: Nice extras

ماژول‌های هدف: `welcome`, `afk`, `vip`, `stats`, `auditLog`

- پیام خوشامد فارسی وقتی کاربر وارد سرور می‌شود با امکان خاموش/روشن کردن.
- ارسال قوانین سرور با پیام خصوصی و نسخه‌بندی متن قوانین.
- سیستم AFK برای انتقال کاربران بی‌حرکت به چنل AFK بعد از زمان قابل تنظیم.
- سیستم VIP برای چنل موقت با ظرفیت بیشتر، زمان بیشتر، priority و template جداگانه.
- آمار روزانه برای ادمین شامل peak آنلاین، تعداد چنل موقت، تیکت‌ها، میانگین پاسخ‌گویی و تخلفات.
- لاگ همه اتفاقات مهم در چنل staff شامل ساخت/حذف چنل، تیکت، بلاک، خطا و عملیات admin.

## معیار پذیرش عمومی

- `npm run build` بدون خطای TypeScript اجرا شود.
- `docker-compose build` با image نهایی non-root و healthcheck فعلی موفق شود.
- هر قابلیت وقتی flag آن `false` است handler، API و job فعال نداشته باشد.
- هر endpoint جدید تست smoke یا سناریوی دستی مستند داشته باشد.
- هیچ پیام انگلیسی برای کاربر نهایی ارسال نشود؛ فقط لاگ‌های فنی می‌توانند انگلیسی باشند.
